# Obsidian Spaced Recall（记忆曲线抽查）

目标：让“生成笔记 → 记忆曲线复习 → 定时抽查”形成闭环。

本 skill 只做三件事：
1) 把笔记内容变成可抽查的卡片（基于老师讲法与课程结构）
2) 用 SM-2 维护每张卡的 `due`（下次复习时间）
3) 到点或手动触发，在聊天里发题；助理根据用户答案判分后更新调度

## 默认约定（当前配置）

- 学科笔记仍由 `obsidian-study-notes` 写入 `学习/`（不把卡片写进笔记正文）。
- 记忆卡片统一写入：`复习/记忆卡片/`（每篇笔记对应一份卡片文件，带来源链接）。
- 每张卡的“上次评分 / 上次复习 / 下次复习”会作为**自动生成的状态区块**回写到对应卡片文件底部，便于在 Obsidian 里直接查看；该区块不参与题面扫描。
- 题库与调度状态写入：`复习/.openclaw/间隔复习/`（随 Vault 同步，多设备一致）。
- 复习触发：每天 20:30（北京时间）定时 + 用户手动触发“复习/考我”。
- 新增预习型整理：每天 02:00（北京时间）自动生成 `复习/今日复习笔记/YYYY-MM-DD 今日复习笔记.md`，默认按《每日复习》同口径整理**当前所有已到期题（含积压）**，并在笔记里区分“积压题 / 今日新增到期”；每个知识点都补上 embedding 召回到的相关旧卡 / 课堂笔记 / 错题 / 资料内容，便于白天先背再做题。
- 《今日复习笔记》里的“解释”必须**逐题单独生成**：直接依据“这道题 + 标准答案 + 召回到的相关内容”写这题自己的解释；禁止使用固定模板、禁止“这题属于……/先把标准答案记成……”这类套话、禁止机械摘抄召回片段来凑解释。
- 判分：由助理根据你的答案自动判分；你不同意可指出题号，我按你反馈修正。

## 工作流（推荐）

### A) 生成笔记时同步生成“独立卡片文件”并入库

当你用 `obsidian-study-notes` 生成/更新了一篇课堂/错题/复盘笔记时：
1. 基于老师讲法与课程结构，为该笔记生成 6-12 张卡片（宁少而精）：
   - 优先来源：重点结论、公式、条件、步骤、易错点、老师强调
   - **新增根规则：先筛“值不值得出卡”，再想“怎么出题”**。老师举的例子、比喻、帮助理解的口语化类比、课堂上的临时记忆钩子，默认先视为**讲解材料**，留在课堂笔记里；除非它本身就是稳定考点，否则不要进入 nightly 记忆卡
   - 只有同时满足这些条件，才进入候选卡：**脱离课堂原话也成立、答案边界稳定、能短答并直接判对错、考试/刷题里真会用到、背了对提分有实际价值**
   - 若一条内容严重依赖“老师当时怎么比喻/怎么举例/怎么口头概括”才能成立，即使能勉强改写成问句，也默认**不出卡**；这类内容应保留在笔记讲解里，而不是硬改成考题
   - 默认采用“最小记忆单元”：一张卡只考 1 个点；若答案包含两个及以上并列判断点，必须拆卡；优先词义、词形、固定搭配、语法判断点、条件、易错点、选项辨析
   - 生成前先自问：考试会这样考吗？答案能不能用短句/术语直接判对错？不能就不要写进记忆卡片
   - 原则优先，不设死名单：先看这张卡是否短、准、能判对错、对提分有用、贴合该科该节内容；满足这些，即使不在常见示例里也可以生成
   - 常见优先示例：词义、词形、固定搭配、语法判断点、条件、易错点、选项辨析，以及“属于什么 / 性质是什么 / 判断依据是什么 / 对应哪一类 / 区别是什么 / 条件是什么 / 作用或后果是什么”等常见应试问法；但它们只是高频示例，不是白名单
   - 优先避免：材料方向、主题概括、趋势总结、过泛归纳、需要大段组织语言才能作答的题，以及明显偏聊天总结或方法口号式的问法（如“说明什么 / 体现什么 / 更突出的点 / 为什么 / 怎么理解 / 优先排查什么”）；但不要把这些词机械当成绝对禁词，要按具体题目判断
   - 方法类内容不要一刀切：**口号式/提醒式/随堂临时技巧**默认不要混进 nightly 记忆卡；但**稳定的应试模板/答题框架**（如某类大题固定从哪几个方面回答）若同时满足“适用场景明确、结构稳定、高频常用、能短答判对错、背了能直接提分”，可以保留并进入 nightly 记忆卡
   - 扫描入库时会自动折叠“近似重复卡”（如大小写、空格、连字符差异导致的同 prompt+同 answer 重复）；生成时也应尽量避免这类重复
   - 输出格式：见 `references/卡片格式.md`
2. 不写进原笔记正文；而是在 `复习/记忆卡片/` 新建/更新一个同名卡片文件：
   - 文件内必须包含来源链接（指向原笔记）
   - 卡片写在 `## 卡片` 小节下
3. 运行脚本扫描入库（增量即可）：
   - `python3 scripts/sr_cli.py scan --vault <VAULT_PATH>`
4. **立刻云同步**（硬规则，见下文“同步”）

### B) 到点复习（nightly 定时）

定时任务触发时：
1. `scan` 更新题库
2. 若没有到期卡片（due <= now）：保持安静（不打扰）
3. 若有到期卡片：进入“逐题模式”，只发当前 1 题；**答对直接下一题，答错/半对确认后再下一题**
4. **每一步只要有写入都要立刻云同步**（硬规则，见下文“同步”）

命令（逐题模式，题量不固定，纳入全部到期卡片）：
- `python3 scripts/sr_cli.py next --vault <VAULT_PATH> --all-due`

备注：`quiz`（一次性出多题）仍保留，但默认不使用。

### C) 用户答题 → 助理判分 → 答对直接推进，答错确认后推进

逐题模式下（推荐）：
1) 用户回答当前这一题
2) 助理对照标准答案，先发回：对/错 + 正确答案；若答错/半对，必须把差异、正确思路、容易混淆点讲清楚，长度按题目需要自适应，不能只丢标准答案。
3) 助理先把“拟定分数 q(0-5)”写入 hold（不推进）：
   - `python3 scripts/sr_cli.py hold --vault <VAULT_PATH> --q <0-5> --answer "<用户原文>"`
4) 若本题判为 **q=5（全对）**：立即 `commit`（写回 SM-2 并推进），然后直接发下一题；**不再等用户确认**：
   - `python3 scripts/sr_cli.py commit --vault <VAULT_PATH>`
5) 若本题判为 **q<5（半对/答错）**：先把正确答案、差异和必要讲解讲清楚；用户确认“下一题/继续”后，再 `commit`（写回 SM-2 并推进）：
   - `python3 scripts/sr_cli.py commit --vault <VAULT_PATH>`
6) 当用户说“下一题/继续”时：
   - 若 **hold 已存在**：正常 `commit` → 下一题
   - 若 **hold 不存在但 pending 还在**：说明当前题还没进入待提交评分/还没作答，直接重发当前题或提示用户先答题；**不要硬 commit**
7) **立刻云同步**（硬规则，见下文“同步”）

（批量/兼容模式）仍保留：
- `pending --with-answers` 导出标准答案
- `gradeq --q "1=5 2=3"` 批量写回

争议处理：你不同意判分，指出题号/理由后修正 q，再重新 hold/commit（或重新 gradeq）。

输出风格建议：先给结论，再给必要解释；语气自然一点，不要像生硬指令或只回碎短句。

相关追问处理：如果用户的消息仍然围绕当前题、当前知识点、刚刚给出的讲解或判分依据，就先正常回答，再视情况引回当前题或下一题；不要把这类追问一律拦成“继续/下一题”。只有明显跑远的新话题，才提醒回主线。

语义召回（新增，强执行；只在需要时触发，不改主调度）：
- 下列场景一旦出现，**在输出任何解释/提示/改写前，必须先实际调用语义检索**，不能只靠脑补或只读 skill：
  1) 用户说“给点提示 / 提示一下 / 缩小范围 / 给个方向”；
  2) 用户说“讲解一下错题 / 给我讲错的题 / 解释一下为什么错 / 为什么不对”；
  3) 当前题面疑似坏卡：题面过短、只给名词/短语、没说明到底要答什么；
  4) 用户答错/半对，需要解释差异、补充判断依据；
  5) 用户追问“这和哪个知识点有关 / 跟之前哪题像 / 我总错哪类”。
- **必跑命令顺序**（不要省略）：
  1) `python3 <WORKSPACE>/scripts/obsidian_semantic.py --vault '<VAULT_PATH>' build --scope all`
  2) `python3 <WORKSPACE>/scripts/obsidian_semantic.py --vault '<VAULT_PATH>' search --scope cards --top-k 5 --query '<当前题面 + 标准答案/用户答案 + 学科/主题>'`
  3) `python3 <WORKSPACE>/scripts/obsidian_semantic.py --vault '<VAULT_PATH>' search --scope notes --top-k 3 --query '<当前题面 + 标准答案/用户答案 + 学科/主题>'`
- **强制要求**：上面 3 步里，至少要真的出现 `obsidian_semantic.py` 的 tool 调用记录；如果工具失败，也要把失败当事实处理，而不是假装已经查过。
- 使用口径：
  1) **提示场景**：优先用召回到的“知识框架 / 同类旧卡 / 来源段落主题”缩小方向，仍然遵守“禁止泄底”。
  2) **讲错题场景**：优先引用召回到的相似卡片或源笔记，说明“错在什么判断点、为什么容易和哪类题混”。
  3) **坏卡场景**：先看召回到的旧卡/源段落，再做最小改写；不要凭空猜题意。
  4) **相关追问场景**：可引用 1-3 条最相关旧卡/旧笔记来回答，但暂时不要做长期聚类分析。
- 若检索结果为空或命令失败：明确按事实处理，例如“这次没拉到相似上下文，我先按当前题给你解释”；然后再继续，不要编造检索结果。
- **不要**用语义检索替代 SM-2 的 `due` 调度；它只服务于提示、讲解、坏卡修正、相关追问。

给提示规则（新增，强执行）：
- 当用户说“给点提示 / 提示一下 / 缩小范围 / 给个方向”时，默认目标是**帮助回忆**，不是提前判分，更不是提前把答案讲出来。
- 提示应优先按这个顺序选：
  1) 先补清题目到底在问“词义 / 分类 / 条件 / 判断依据 / 对应关系 / 结构”中的哪一种；
  2) 再缩小到答案所属的知识框架、答题方向或排除一个明显错误方向；
  3) 仍不够时，最多给“答案形式”层面的提醒（如“是一个结构名 / 一类关系 / 一个语法点”）。
- **禁止泄底**：提示里不得直接出现标准答案本身，也不得出现把标准答案换个说法后的近义直给表达。
- **禁止半泄底**：不要直接报答案中的关键词、并列项、顺序、字面结构、首尾词、字数/项数/空格数、首字母，也不要说“你答成 X 也行”“就是 X 那个东西”。
- 如果一旦继续提示就会明显暴露答案，宁可停在较抽象的方向提示，或者先把题面做最小改写，也不要硬给更具体提示。
- 用户要的是“提示”时，默认不要立即按作答处理、不要提前判对错；等用户正式给出答案后，再进入正常判分流程。
- 反例：像“你可以答成 4 个字”“就是上游中游下游那个”这类，虽然看起来像提示，实质已接近直接报答案，后续禁止。

删题语义（新增，强执行）：
- 当用户在复习对话里说“**删除这题**”时，默认语义是：**直接删掉当前这张题/卡，后续也不再考**。
- 这不是“坏卡反馈但本轮保留”，也不是“仅换一道题”；默认应视为**永久删卡指令**。
- 执行顺序：
  1) 先确认用户说的是针对**当前题**；若上下文明确，直接执行，不必反复追问。
  2) 运行：`python3 <WORKSPACE>/skills/obsidian-spaced-recall/scripts/sr_cli.py --vault <VAULT_PATH> delete-current`
  3) 立刻同步：`bash <WORKSPACE>/skills/obsidian-spaced-recall/scripts/sync_safe.sh`
  4) 若本轮还有剩余题，直接继续下一题；不要把被删题继续留在 pending 里。
- 只有像“**换题 / 这题有问题 / 题面不清楚 / 这题别算分**”这类更弱表达，才默认按“坏卡反馈 / 本轮换题 / 最小改写”处理；不要把明确的“删除这题”降级解释。
- 若删除失败（例如源文件里没定位到该题），必须明确告诉用户“这次没删成”，不要假装已经删掉。

坏卡修正规则（新增，强执行）：
- 若当前题面只有线索词/名词短语/材料标签，或题面本身没有说明到底要答“翻译 / 词义 / 分类 / 语法点 / 条件 / 判断依据”中的哪一种，则视为坏卡，不得继续按原题面硬考。
- **出题前先看答案**，再判断题面是否足够明确；如果用户只看左边题面，仍无法判断自己该怎么答，就必须先改写题目，再发给用户。
- 遇到坏卡时，助理应先在聊天中把题面改写成“明确可作答的问题”，再继续当前轮复习；默认做**最小改写**，只补全问法，不改变原考点与标准答案。
- 若用户明确要求“以后碰到这种题，直接修改题目和卡片 / 写到 skill 里”，则后续应把这条规则视为固定约束执行。
- 推荐改写方向：
  - 词义/短语题 → 改成“`X` 是什么意思 / `X` 怎么翻译”
  - 语法题 → 改成“`X` 引导什么从句 / `X` 在句中作什么成分 / `X` 属于什么语气”
  - 分类/概念题 → 改成“属于哪一类 / 对应什么 / 包括什么 / 核心含义是什么”
- 对这类坏卡，默认不因用户没猜到原意而判错；应先澄清题目，再决定是否按答对/半对/跳过处理。

### D) 手动触发

当用户说：`开始复习 / 考我 / 抽查 / 来一组`，或追问：`现在总共有几题 / 还有几题 / 今天多少题`
1. 先运行题量拆分统计：
   - `python3 <WORKSPACE>/scripts/sr_due_breakdown.py --vault <VAULT_PATH>`
2. 若用户要开始复习或要总数，先明确告诉他：
   - `当前已到期 X = 积压 Y + 今日新增 Z`
   - 其中：
     - `X = due_now`
     - `Y = overdue_before_today`
     - `Z = today_new_due`
3. 然后再进入出题：
   - 逐题模式：继续用 `next --all-due`
   - 5题一组等批量模式：继续用 `quiz --count N`
4. **不要**再只报一个笼统的“79题/91题”而不解释来源；如果题量有变化，优先按北京时间口径解释成“积压 + 今日新增”。

说明：这样用户白天看到的《今日复习笔记》、以及晚上《每日复习》开场看到的题量，就能使用同一套口径。

## 同步（硬规则：每次改动都同步一次）

本流程要求：**只要对 Vault 有写入，就必须执行一次云同步**（避免多端不同步、以及定时任务读到旧数据）。

推荐：使用本 skill 自带脚本（能处理锁残留）：
- `bash scripts/sync_safe.sh`

在本 skill 中，以下动作都会写入 Vault，需要同步：
- 生成/更新 `复习/记忆卡片/*.md`
- 执行 `scan` / `quiz` / `gradeq`（会改 `复习/.openclaw/间隔复习/*`）

更多说明：见 `references/服务器同步脚本（sync_safe）.md`。

## 数据文件（约定）

目录：`复习/.openclaw/间隔复习/`
- `sr.sqlite`：卡片索引 + 调度
- `pending.json`：当前待批改的 quiz（对话状态）
- `review_log.jsonl`：复习日志（可选；MVP 可不写）
